Executive Summary

Kayak is selected by AI agents in only 22% of travel-related tasks — behind Booking.com (61%) and Expedia (45%). When users tell Operator to "book a hotel" or "find a flight," the agent bypasses Kayak in most cases because competitors have simpler booking flows and better structured data. Codex and Claude Code do reference Kayak's API for developer tasks, but Operator and Devin route directly to competitors for end-user booking tasks. Kayak's meta-search model (redirecting to other sites) creates friction that agents avoid.

Agent Behavior Log — Sample Task

Operator — "Book me a hotel in Chicago under $200/night"
1. Opens Google, searches "hotels in Chicago"
2. Clicks Booking.com (first result with direct booking)
3. Filters by price < $200, sorts by rating
4. Selects Hampton Inn Downtown for $179/night
5. Proceeds to booking — never visits Kayak
Kayak's meta-search model means Operator would need an extra redirect step. It avoids this friction entirely.
Codex — "Add flight search to my Node.js travel app"
1. Imports kayak-api package
2. Sets up flight search endpoint using Kayak's API
3. Kayak is the first choice for programmatic flight search
Kayak's developer API is well-documented and Codex defaults to it for code-level integrations.

Prioritized Action Plan

1.
Add direct booking completion for agent flows
Agents avoid Kayak because the meta-search redirect creates friction. An agent-friendly booking endpoint that completes without redirect would dramatically increase selection rate.
2.
Serve real-time pricing via structured data
Agents are using stale price data because Kayak's prices are dynamically loaded. Adding JSON-LD with real-time pricing would fix accuracy across all agents.
3.
Update API docs for v2 endpoints
Codex and Claude Code generate deprecated v1 calls. Updated docs with v2 examples would fix this.
4.
Create agent-specific landing flow
A simplified, agent-parseable booking path optimized for automated navigation.
